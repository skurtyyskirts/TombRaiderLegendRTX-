================================================================================
1. ARCHITECTURE OVERVIEW
================================================================================

LA Noire uses a DUAL DX9/DX11 rendering engine with an AbstractDevice abstraction layer.

Key RTTI classes discovered:
  - AbstractDevice      (0x013DE640) - Base device interface
  - AbstractDevice9     (0x013DF93C) - DX9 backend
  - AbstractDevice11    (0x013DF7B4) - DX11 backend
  - AbstractThreadedDevice (0x013DE660) - Threading wrapper
  - DX9_Engine          (0x013DE604)
  - DX9RenderTexture    (0x013DE4F8)

The engine does NOT import SetVertexShaderConstantF/SetPixelShaderConstantF directly.
All D3D9 device calls go through the AbstractDevice COM vtable wrapper.

DX9-related imports (d3d9.dll):
  - Direct3DCreate9
  - D3DPERF_BeginEvent
  - D3DPERF_EndEvent

DX9X imports (d3dx9_43.dll):
  - D3DXCompileShader    (used at 6 call sites)
  - D3DXSaveSurfaceToFileA

DX11 imports also present:
  - D3D11CreateDevice
  - D3DX11CompileFromMemory

Shader programs loaded from: "out/graphicsdata/programs.vfp.dx9" (DX9 path)
                              "out/graphicsdata/programs.vfp.dx11" (DX11 path)

================================================================================
3. SHADER CONSTANT HANDLE FORMAT
================================================================================

The engine uses a PACKED 32-bit handle for each named shader constant:

  Bits 24-31: Number of float4 registers (count)
  Bits  8-15: Vertex Shader register index (0xFF = not used in VS)
  Bits  0-7:  Pixel Shader register index (0xFF = not used in PS)

Handles are obtained via ID3DXConstantTable::GetConstantByName (vtable offset 0x1FC)
during renderer initialization at 0x00D58E88.

The constant table object is accessed from the renderer via **(*renderer + 0x1FC).

================================================================================
4. SetVertexShaderConstantF MECHANISM
================================================================================

The engine does NOT call IDirect3DDevice9::SetVertexShaderConstantF directly.
Instead, it uses a custom abstraction:

CONSTANT SETTING WRAPPER: fcn.00D60AB0
  Signature: void __thiscall SetShaderConstant(uint32_t handle, void* data)
  - ECX = packed handle (see format above)
  - [esp+4] = pointer to float data

  Implementation:
    1. Reads render context from TLS: FS:[0x2C] -> [slot0] -> [+8] -> [ptr]
    2. Gets vtable of the render context object
    3. If VS register != 0xFF: calls vtable[5] (offset 0x14) = SetVSConstant(reg, data, count)
    4. If PS register != 0xFF: calls vtable[6] (offset 0x18) = SetPSConstant(reg, data, count)

  This function has 295 xrefs (call sites) throughout the engine.

The render context vtable methods:
  [5] offset 0x14 = SetVertexShaderConstant(int startRegister, void* data, int count)
  [6] offset 0x18 = SetPixelShaderConstant(int startRegister, void* data, int count)

These methods are virtual and have DX9 and DX11 implementations.
The DX9 implementation internally calls IDirect3DDevice9::SetVertexShaderConstantF
through the raw D3D9 COM vtable.

Device wrapper vtable methods (on renderer[0x5b8] object):
  offset 0x04 = SetRenderState(state, value)
  offset 0x0C = SetSamplerState(stage, type, value)
  offset 0x14 = SetVertexShaderConstant(reg, data, count)
  offset 0x18 = SetPixelShaderConstant(reg, data, count)
  offset 0x28 = SetTexture(stage, texture)
  offset 0x38 = ??? (draw-related)
  offset 0x48 = ??? (present/flush)
  offset 0x4C = CreateVertexBuffer?
  offset 0x50 = CreateIndexBuffer?
  offset 0x60 = CreateTexture?
  offset 0x64 = CreateRenderTarget?
  offset 0x6C = SetVertexDeclaration?
  offset 0x70 = SetRenderTarget?
  offset 0x7C = ??? (present)
  offset 0xBC = DrawPrimitive?

7. SetTransform USAGE (FFP)
================================================================================

The engine does NOT use IDirect3DDevice9::SetTransform for fixed-function pipeline.
All transforms are passed via shader constants through the abstraction layer.
The D3DRS_* render state values set in SetDefaultRenderStates confirm the engine
uses the programmable pipeline exclusively.

No "SetTransform" string found in the binary.

================================================================================
8. D3DX CONSTANT TABLE vs DIRECT REGISTER WRITES
================================================================================

The engine uses a HYBRID approach:

During INITIALIZATION (0x00D58E88):
  - Uses ID3DXConstantTable::GetConstantByName (vtable 0x1FC) to resolve
    named constants to packed register handles
  - The constant table comes from D3DXCompileShader output

During RENDERING (0x00D60AB0 + 295 call sites):
  - Uses DIRECT REGISTER WRITES via the AbstractDevice wrapper
  - The packed handle already contains the start register and count
  - Does NOT call ID3DXConstantTable::SetMatrix/SetVector at render time
  - Directly calls AbstractDevice::SetVSConstant(startReg, data, count)
    which wraps IDirect3DDevice9::SetVertexShaderConstantF

This is efficient: resolve names once at init, then use direct register writes per-frame.

================================================================================
9. TLS RENDER CONTEXT CHAIN
================================================================================

The constant setter reads the active render context from Thread-Local Storage:

  FS:[0x2C]     -> TLS slot array
  [slot 0]      -> Module TLS data
  [TLS + 8]     -> Pointer to active render context pointer
  *[that ptr]   -> Active render context (AbstractDevice instance)
  [context + 0] -> vtable
  [vtable+0x14] -> SetVertexShaderConstant
  [vtable+0x18] -> SetPixelShaderConstant

This TLS indirection allows different threads to have different render contexts,
supporting the threaded rendering architecture.
