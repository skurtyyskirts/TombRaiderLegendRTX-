# Builds 074–077 — The Asset Pipeline Breakthrough

> Four builds. The single most consequential discovery in the entire project archive. **`user.conf` in the game directory had `rtx.enableReplacementAssets = False` silently overriding `rtx.conf`'s `True` — for 58 consecutive builds (016–074).** Plus the cold-launch crash fix.

This epoch's lesson is the dominant one: **runtime configuration override files matter as much as the canonical config**. Anyone porting another game to RTX Remix must learn this story first.

## Build 074 — Deferred patches, proxy hardening

Pure proxy hardening. No new culling patches.

### What changed
- **Deferred `TRL_ApplyMemoryPatches()`** to the first `BeginScene` where `viewProjValid=1`. Previously patches landed during device creation, before the renderer was initialized. This eliminated a menu crash (and removed the dependency on `TR7.arg`, the launch flag that skipped the menu to avoid the crash).
- **Permanent page unlock.** A single `VirtualProtect(PAGE_READWRITE)` for all 8 data pages used by per-scene stamping — previously the proxy locked/unlocked per frame, adding cost and risk.
- **Release ordering fix** in proxy cleanup.
- **DIAG gating** — diagnostic log lines moved behind `DIAG_ENABLED` so a future perf build can strip them.

### Result
All 31 patches applied. 3,749 draws per scene. `mod.usda` hashes flagged as stale (suspected — confirmed in build 075).

The build 074 SUMMARY.md is the first to explicitly call out the staleness hypothesis: "the 8 anchor mesh hashes in `mod.usda` were captured under a previous Remix configuration, likely before `positions` was added to the hash rule and before SHORT4→FLOAT3 expansion."

## Build 075 — The breakthrough

The configuration override that wasted two months of testing.

### The discovery

While preparing for a fresh Remix capture to update mod.usda, the test agent dumped every config file in the game directory. Found:

```ini
# user.conf (game directory)
rtx.enableReplacementAssets = False
```

But `rtx.conf` in the same directory had:

```ini
# rtx.conf
rtx.enableReplacementAssets = True
```

**`user.conf` wins.** It is generated by the in-game Remix menu and overrides the project's canonical `rtx.conf`. With `enableReplacementAssets = False`, every replacement mesh, every anchor light, every USD-defined override is silently ignored.

This had been the case since build 016, when `enableReplacementAssets = True` was added to the project's `rtx.conf`. Some prior session had opened the Remix menu (X key, `remixMenuKeyBinds = X`), toggled the replacement-assets switch off, and saved — generating `user.conf` with the negated value. From that point on, every subsequent build was testing with replacement assets disabled.

**58 builds (016–074) were not actually testing what their authors thought they were testing.**

### The proof

After fixing `user.conf` to `True`, the test added a single purple light anchored to `mesh_574EDF0EAD7FC51D` (a known-stable mesh). Tested light radius 2, 5, 500, 3000 — no functional change beyond expected radius behavior.

**Result:** Purple light visible in all 3 hash-stability screenshots. The light shifts position with camera motion (proving the anchor is working, not just a fallback). Replacement asset pipeline confirmed end-to-end.

### The reframe

The 8 building hashes in `mod.usda` are now confirmed stale. Geometry IS rendering (3,749 draw calls per scene). But no rendered mesh matches the stored hashes. Three years of building-light work were against a `mod.usda` whose anchor list was obsolete the moment `positions` was added to the hash rule.

The white-dot artifacts from builds 072–073 are confirmed as **denoiser/NRC artifacts** — they don't scale with light radius.

### The dead end logged

[[Dead-Ends]] #14:

> `user.conf` overriding `rtx.enableReplacementAssets=True`: `user.conf` had `enableReplacementAssets=False` generated by in-game Remix menu; silently disabled all mod content builds 016–074 — fixed build 075.

### The CPU smooth normals attempt

Also in this build window: the proxy attempted CPU smooth normals (stream 1 FLOAT3 injection). Built and compiled but didn't work at runtime. Changes the geometry descriptor hash of every mesh; possible IB lock / stream 1 / Remix capture conflicts. Abandoned in favor of `rtx.smoothNormalsTextures` (Remix-side, no proxy changes needed). ## Build 076 — Crash protections restored

Build 075 was so big it accidentally regressed two crash protections. Build 076 restored them.

### What was restored
- `patch_null_crash_40D2AF()` — code-cave trampoline at `0x40D2AC` guarding NULL+0x20 deref at `0x40D2AF`
- PUREDEVICE stripping — `behFlags &= ~0x00000010` in `W9_CreateDevice` (Remix cannot see pure-device state)
- FourCC format rejection — `D3DERR_NOTAVAILABLE` for `cf > 0xFF` (avoids a hang when the game probes for compressed formats Remix doesn't support)
- `build.bat` VS 18 Community fallback (previously required Insiders)

### Result
**Hash stability PASS** — consistent colors across pans, green floor stable. Clean render captured in transition area; baseline test confirmed purple light still visible. **3,733 draws/scene crash-free.**

This is the build that re-established forward progress momentum after the build-075 reframe.

## Build 077 — The cold-launch crash fix

Builds 001–076 all passed in automated testing, but only because `TR7.arg` was used to skip the main menu directly to the level. **A cold launch from the main menu** (the natural user flow) crashed at NRC initialization with `STATUS_STACK_BUFFER_OVERRUN (0xc0000409)` inside `d3d9_remix.dll+0x001654dc`.

### Root cause

`DrawCache_Record()` was storing un-referenced COM pointers (`vb`, `ib`, `decl`, `tex0`). When NRC's first raytrace frame replayed menu-geometry draws, the cached pointers had been released and freed by the game's own state-cleanup between the menu and the level. Use-after-free.

### The fix

`AddRef` on record, `Release` on eviction. Standard COM ownership pattern.

### Why it took until build 077

Every test in builds 001–076 used `TR7.arg`. The menu path was never exercised in automation. Manual cold launches by the user occasionally crashed, but the crashes were attributed to other causes (driver issues, the per-build instability that came with each new patch). The static + dynamic analysis combination that finally caught this was:

1. Cold launch crash repro → minidump capture
2. `dumpinfo.py diagnose --binary d3d9_remix.dll` identifies the crash site as a stack-buffer-overrun inside Remix
3. `throwmap.py match` cross-references against `_CxxThrowException` sites
4. Static-analyzer subagent decompiles the crash function — identifies it replays cached draws
5. Decompile DrawCache record/eviction path in the proxy — `AddRef` missing
6. Patch + verify with `--filter event==COLD_LAUNCH` on the dx9tracer

### Result
Game survives 90+ seconds in menu → Peru transition. 2,468 draws per scene with all 31 patches active. Cold launch officially stable.

This build also formalized the **mandate to test cold launches**, not just `TR7.arg` launches.

## What this epoch established

- **Configuration override files are real.** Any port that uses Remix must check both `rtx.conf` and `user.conf` in the game directory, and ideally any other config the in-game menu can write. A `user.conf` audit is now part of the standard build template.
- **mod.usda anchor hashes go stale.** Every change to `rtx.geometryAssetHashRuleString`, every change to vertex format pre-processing in the proxy, invalidates the entire anchor list. A fresh capture must follow every hash-rule or vertex-pipeline change.
- **Fallback lights are dangerous diagnostic tools.** Builds 019–037 spent a lot of time chasing a fallback light. Always test with neutral white fallback during diagnostics.
- **Cold launches must be tested.** `TR7.arg` is a useful test convenience but hides at least one class of bug (use-after-free between menu and level state).
- **`AddRef` everything cached.** Standard COM, but easy to forget when writing a proxy that records and replays draw state.
- **`patch_null_crash_40D2AF()` and PUREDEVICE stripping are non-negotiable.** Removing either crashes the proxy. They are now permanent fixtures in `proxy/d3d9_device.c`.

## Open issues at end of epoch

- The 8 anchor mesh hashes in `mod.usda` are stale. **Fresh Remix capture is the next action.** The pipeline is proven working — only the hash data is wrong.
- Skinned character hash drift (Lara, distant NPCs). World geometry is stable but skinned meshes flicker between frames. Investigation continues in build 079.
- Performance — the DIAG-disabled build (078) is the first opportunity to measure the proxy's actual frame cost.
